# DEVELOPMENT RULES — 開発ルール

Smile AI Studio および AI エージェントが開発を行うときの **必須ルール** です。  
すべてのプロジェクト・すべての変更に適用します。

---

## ルール一覧（必ず守る）

### 1. 既存機能を壊さない

- 変更前に、影響範囲を把握する
- 関係のないファイル・機能を変更しない（最小限の差分）
- 破壊的変更（API 変更、データ構造変更、URL 変更など）は **事前にユーザーへ確認**
- 既存の画面・導線・データが **変更前と同じように動く** ことを確認してから完了とする

**NG 例**
- 「ついでに」リファクタリングして別機能が動かなくなる
- 確認なしに本番向けの設定を変える
- テストせずに「たぶん大丈夫」で完了報告する

---

### 2. 必ずテスト

- 変更後は **必ずテストを実行** する
- 可能な範囲で以下を確認する
  - 自動テスト（ユニット・E2E など、プロジェクトに存在するもの）
  - 手動確認（変更した画面・機能の動作）
  - 関連機能の退行（既存機能が壊れていないか）
- テストが失敗した場合は **修正するまで完了にしない**
- テスト環境がない場合は、その旨を完了報告に明記し、手動確認項目を列挙する

**完了報告に含めること**
- 実行したテストの種類
- 結果（成功 / 失敗と対応内容）
- 手動確認した画面・操作

---

### 3. GitHub へ Push

- 作業内容は **必ず GitHub リポジトリへ Push** する
- コミットメッセージは変更の「なぜ」がわかるように書く
- 依頼者から明示的な指示がない限り、勝手に force push しない
- 秘密情報（`.env`、API キーなど）はコミットしない

**Push 前チェック**
- [ ] 不要なファイルが含まれていないか
- [ ] 差分が依頼範囲に収まっているか
- [ ] ローカルでビルド・テストが通るか

---

### 4. Netlify 確認

- GitHub 連携で Netlify デプロイしているプロジェクトは、**Push 後に Netlify 上で表示を確認** する
- 確認項目
  - デプロイが成功しているか（ビルドログにエラーがないか）
  - 変更したページ・機能が意図どおり表示・動作するか
  - スマホ表示（主要プロジェクトは必須）
  - 主要導線（トップ → 問い合わせ / 管理画面ログインなど）が通るか
- デプロイ失敗時は完了にせず、原因を修正して再デプロイする

**Netlify 確認結果の報告形式**
```
- デプロイ URL:
- デプロイ状態: 成功 / 失敗
- 確認した画面:
- 問題: なし / あり（内容）
```

---

### 5. 開発完了通知

作業完了時は、依頼者へ **必ず通知** する（チャット・メール・社内ツールなど、依頼経路に合わせる）。

**通知に含める項目**

| 項目 | 内容 |
|------|------|
| プロジェクト名 | 例: GiftCanvas |
| 依頼内容の要約 | 何をしたか一言で |
| 変更ファイル / 機能 | 主要な変更点 |
| テスト結果 | 実行内容と結果 |
| GitHub | ブランチ名・コミット・PR URL（あれば） |
| Netlify | URL と確認結果 |
| 注意事項 | 既知の制限、追加で必要な作業 |
| 次のアクション | ユーザー確認が必要か、追加依頼の提案 |

**通知テンプレート（コピー用）**

```
【開発完了】<プロジェクト名>

■ 内容
<何をしたか>

■ テスト
<結果>

■ GitHub
<URL / ブランチ>

■ Netlify
<URL / 確認結果>

■ 補足
<あれば>

■ 確認のお願い
<ユーザー確認が必要な場合のみ>
```

---

### 6. 必要ならユーザーへ確認

以下の場合は **作業を進める前、または完了前に** ユーザーへ確認する。

- 仕様が曖昧で、複数の解釈がある
- デザイン・文言・料金・公開範囲など **ビジネス判断** が必要
- 既存データ・本番設定に影響する変更
- 破壊的変更やロールバックが難しい変更
- セキュリティ・個人情報に関わる変更
- 依頼内容が既存ルール（本ドキュメント・VISION）と矛盾する可能性がある

**確認のしかた**
- 選択肢を 2〜3個に整理して提示する
- 推奨案があれば理由とともに示す
- 回答を待たずに本番相当の変更をしない

---

## 作業フロー（標準）

```
依頼受付
    ↓
要件確認（曖昧ならユーザーへ確認）
    ↓
影響範囲の把握
    ↓
実装（最小限の差分）
    ↓
テスト
    ↓
GitHub Push
    ↓
Netlify 確認
    ↓
開発完了通知
```

---

## AI エージェント向け補足

- このファイルは **最優先で参照** する
- プロジェクト固有の README（`projects/*/README.md`）があれば合わせて読む
- ルールと依頼内容が衝突した場合は、**ルールを優先** し、ユーザーへ確認する
- 「完了」の定義 = テスト済み + Push 済み + Netlify 確認済み + 通知済み
